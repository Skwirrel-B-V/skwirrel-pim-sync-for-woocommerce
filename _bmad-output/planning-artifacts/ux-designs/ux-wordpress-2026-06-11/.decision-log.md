## Key UX decisions & assumptions (fast-path — confirm at review)
1. **Reuse the existing `.skw-dashboard` design system for ALL new surfaces.** No new visual language. DESIGN.md = distillation of `dashboard.css`.
2. **Sync-state UI must evolve from 7 phases → resumable per-product progress** (architecture replaces the phased orchestrator with the per-entity-atomic ledger). Progress shows "X of Y items committed · resumable", plus a paused state. `[ASSUMPTION]`
3. **Preflight (FR-16) is a STEP in the Sync-Now flow, not a separate destination** — "Preview what will change" before "Commit". Renders the Change_Set: product counts (added/changed/removed, clickable → FR-15 deep-links) AND a category-structure block (created/renamed/removed-or-orphaned + re-homed counts). `[ASSUMPTION]`
4. **Health / Diagnostics is a NEW top-level action block** → its own screen: overall verdict (healthy/warning/problem) + per-check list w/ plain-language + fault attribution (ours/environment) + conflicts + "Copy report for support". `[ASSUMPTION]`
5. **FR-17 reset = formalize the existing Danger Zone** with a preflight-style "this will remove N products" preview + explicit typed/confirmed opt-in; trash (recoverable) default. `[ASSUMPTION]`
6. **Guided setup (FR-1) replaces the hub on first run** with an ordered flow: Connect token → Verify live → Choose essentials (sensible defaults pre-filled) → First sync (offer preflight). Dismissable, never returns once complete. `[ASSUMPTION]`
7. **Setting relations (FR-3):** inactive settings render visibly disabled with a one-line "Inactive because {setting} is off" reason, grouped by intent (Connection · What to sync · How it looks · Advanced). `[ASSUMPTION]`
8. **There IS vanilla JS already** (log polling, modal, abort, auto-refresh) — "no JS framework" ≠ "no JS". Preflight/health can use the same lightweight AJAX patterns; no React/Vue. `[ASSUMPTION]`
9. **Fault attribution & conflicts in plain language for non-technical owners** — verdict first, evidence/technical detail behind a "Details" disclosure; never a stack trace as primary output. `[ASSUMPTION]`

## Confirmed by Jos (2026-06-11) — four decisions
1. **Settings intent-grouping (FR-3) = four groups:** **Connection** (token + endpoint) · **What to sync** (categories/brands/grouped/selections/collections) · **How it looks** (images, slugs/permalinks, language) · **Advanced** (timeout, retries, batch size, purge, verbose). Relation-disabled fields dim + show "Inactive because {setting} is off".
2. **Sync progress = per-item resumable:** a single progress representation showing **"X of Y products · resumable"** (committed/total, tabular-nums) + current activity + Resume affordance. **Retires the 7-phase banner.**
3. **Reset (FR-17) confirm = Preview + checkbox opt-in:** show the "removes all N Skwirrel products (to Trash, recoverable)" change-set preview, then a required "I understand" checkbox + danger button. (No typed-confirm; no permanent-delete variant this chapter.)
4. **Health verdict badge on the hub block:** the Health & Diagnostics action block carries a standing healthy/warning/problem badge from the last check (re-runs on open), so problems surface at a glance.
